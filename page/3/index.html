《构建之法》笔记一

发表于 2015-10-09   |   分类于 读书心得   |  

构建之法

从知乎看到很多人推荐这本书，便在多看买了本电子版。翻了几页就完全被这本书折服，的确可以被评为****软件开发必读经典****。作者邹欣老师在微软工作有着多年的软件开发经验，不同于学院派的教学方法，其主张“学中做”的学习方法确实有效的让学生掌握了知识。

精华摘要

  • 构建不仅仅是cc和link命令，一个复杂的软件不但要有合理的软件架构（Software Architecture）、软件设计与实现（Software Design, Implementation and Debug），还要有各种文件和数据来描述各个程序文件之间的依赖关系、编译参数、链接参数，等等。这些都是软件构建的过程。
  • 软件团队要从需求分析（Requirement Analysis）开始，把合适的需求梳理出来，然后逐步展开后续工作，如设计（软件架构）、实现（写数据结构和算法）、测试，到最后发布软件。
  • 一个扩展的推论是：
    软件企业 = 软件 + 商业模式

  • 程序（算法、数据结构）是基本功，但是在算法和数据结构之上，软件工程决定了软件的质量；商业模式决定了一个软件企业的成败。软件从业人员和软件企业的道德操守会极大地影响软件用户的利益。

  • 软件工程是把系统的、有序的、可量化的方法应用到软件的开发、运营和维护上的过程。

  • 软件工程包括下列领域：软件需求分析、软件设计、软件构建、软件测试和软件维护。

  • 软件工程和下列的学科相关：计算机科学、计算机工程、管理学、数学、项目管理学、质量管理、软件人体工学、系统工程、工业设计和用户界面设计。

  • 哲学家的宗旨是：我思，故我在
    科学家的宗旨是：我发现，故我在
    工程师的宗旨是：我构建，故我在

懒散的态度

发表于 2015-03-30   |   分类于 人生感悟   |   
从28号到30号，几乎没有认真看过一下书，骨子里那种惰性再次散发出来。

        29号下午，nchu的学长搞了一个nchu的ACM新手选拨赛，闻讯我赶去围观。一点钟和大家一起开始做题，总共10道题。于是从第一题开始看，看完题目感觉题目意思还是很容易理解，便着手写代码，写完感觉良好再和例子一对没什么问题，然而提交之后一直TL，又修修改改大概10分钟都不知道错在哪儿了，便放弃了。接着看了下榜发现有道题很快被人解决了，马上看那到题，一看发现真的很简单。但是这时候脑子又抽了，一个用最大值减最小值再除以2的问题，硬是用sort函数加数组解决；结果又是一个TL，关键是一直没想到正确的改法，改了3次一直TL，又放弃了。继续下一题，一个简单的斐波那契，学过的都懂，作死用了递归又是一个TL，幸好马上改成迭代，终于过了一道题。此时过去了一个小时，接着第四题题目意思还是很简单，求AK得最大概率，直接想当然的把每道题的最大概率相乘，果不其然虽然和题目给的输入输出对上了，却依然错了。又卡在这上面。就继续转往下一题，一个求素数和的简单问题，结果又是一个TL。这个时候我的耐心彻底被消耗完了，把每道题都看了一遍，发现2道完全不会，剩下的都有思路但是细节很多需要仔细考虑。在多次TL，WA后放弃写题。玩了一下午FIFA。晚上看了群里发出来的题解发现自己的很多错误，并且忽略了很多东西，把题目简单化想当然。哎，究其根本还是题目写的太少看的太少，至今题量还在70+；还大部分水题。懒散的态度无疑成为我前进的拦路虎，不解决的话无论学什么都不可能成功。

无论怎么样，既然把ACM作为我大学的目标，我就要付出很多努力。不想到了大二大三后悔现在题目做的少。

从4月份开始，每天至少2小时切题。争取暑假前切掉400道题以上，为了前路加油！！！！
